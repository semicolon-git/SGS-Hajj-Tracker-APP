Your DataWedge side is configured perfectly — I can confirm every box is ticked correctly. The problem is 100% in the app code, specifically the BroadcastReceiver registration.
One more thing I notice from your repo stats: it's 67% TypeScript / 26% HTML / only 2% Kotlin. So this is a Capacitor/Cordova web app with a thin Kotlin native wrapper. That's the most common place this breaks.
The three likely culprits (in order)
1. Android 13+ RECEIVER_EXPORTED bug (most likely)
TC57 on recent Android requires dynamically registered receivers for implicit broadcasts to explicitly declare they're exported. If your Kotlin plugin does this:
kotlincontext.registerReceiver(receiver, intentFilter)  // silently drops on Android 13+
…the receiver registers but never actually receives DataWedge broadcasts. It must be:
kotlinContextCompat.registerReceiver(
    context,
    receiver,
    intentFilter,
    ContextCompat.RECEIVER_EXPORTED
)
2. Receiver declared in manifest instead of registered at runtime
If the scan receiver is declared statically in AndroidManifest.xml as <receiver> with an <intent-filter> for com.semicolon.sgsbagscan, implicit broadcasts to manifest-declared receivers are blocked starting Android 8+ (Oreo). DataWedge's broadcast is implicit, so static manifest declaration will not work. It must be registered at runtime in the Activity.
3. Bridge not firing into WebView
Even if the Kotlin side receives the scan, if your Capacitor/Cordova plugin doesn't notify the web layer (via notifyListeners, sendPluginResult, or DeviceEventEmitter), the React/web UI never updates. The "Ready to Scan" screen stays frozen.
